## Task 1: Introduction

Wazuh is an all-in-one open-source security solution focused on threat detection, integrity monitoring, vulnerability assessment, and log analysis. It follows a classic **manager-agent** architecture.

Key points I noted:
- The **agent** is the lightweight software installed on endpoints to monitor activity and forward logs.
- The **manager** is the central server that collects, analyzes, and visualizes everything.

**Questions & Answers:**
1. When was Wazuh released? → **2015**
2. Term for a monitored device? → **Agent**
3. Term for the central managing device? → **Manager**

## Task 4: Wazuh Vulnerability Assessment & Security Events

Wazuh automatically scans agents for vulnerabilities by checking installed software versions against known CVEs. Scans happen on agent deployment and then periodically.

Security Events are rule-based alerts. Some can be noisy (e.g., normal file deletions), so filtering by time, severity, or MITRE tactics is very useful.

## Task 6: Monitoring Logons with Wazuh

Focused on detecting successful and failed logon attempts (SSH, RDP, Windows logons, etc.).

Failed SSH attempts often trigger Rule ID 5710. Alerts include rich context like source IP, username, and MITRE mappings. Logs are stored in /var/ossec/logs/alerts/alerts.log.

This is a core use case for any EDR — catching brute-force or unauthorized access early.

## Task 7: Collecting Windows Logs with Wazuh

Windows logs are ingested using **Sysmon** for detailed visibility beyond standard Event Viewer.

**High-level steps:**
- Install and configure Sysmon (custom XML for things like PowerShell execution)
- Update the Wazuh agent's ossec.conf to monitor the Sysmon channel
- Restart services

**Questions & Answers:**
1. Tool used to monitor detailed system events? → **Sysmon**
2. Standard Windows app that records system events? → **Event Viewer**

## Task 8: Collecting Linux Logs with Wazuh

Linux log collection is straightforward and similar to Windows. We configured the Wazuh agent to forward **Apache2 web server** logs using the built-in 0250-apache_rules.xml ruleset.

Wazuh ships with \~900+ rulesets for common apps (Docker, FTP, WordPress, SQL Server, MongoDB, Firewalld, etc.). You can also create your own.

The configuration goes in the agent's /var/ossec/etc/ossec.conf file using a <localfile> block.

## Task 9: Auditing Commands on Linux with Wazuh

Wazuh integrates nicely with **auditd**, the Linux Auditing System, to monitor command execution, file access, privilege escalation attempts, and other critical system events.

In this task, I installed auditd on the Linux agent, configured custom audit rules, and let Wazuh forward those logs for analysis and alerting. This gives deep visibility into what commands are being run on Linux endpoints.

**Key Steps I Followed:**
1. Installed auditd on the Linux agent (works on both Debian/Ubuntu and CentOS/RHEL).
2. Added monitoring rules in /etc/audit/rules.d/audit.rules.
3. Ensured the Wazuh agent was configured to collect and forward the auditd logs to the manager.

**Questions & Answers:**
1. What application do we use on Linux to monitor events such as command execution?  
   **Answer:** auditd

2. What is the full path & filename for where the aforementioned application stores rules?  
   **Answer:** /etc/audit/rules.d/audit.rules

## Task 10: Wazuh API

This task introduced Wazuh’s powerful REST API, which is excellent for automation, scripting, and integrating with other tools.

I learned that you can interact with the manager directly using standard tools. The dashboard also has a built-in **API Console** under Tools, which makes testing easy.

**Key Points:**
- Standard Linux tool for making requests: *curl*
- Use **GET** method to retrieve information
- Use **PUT** (and other methods) to perform actions
- API is very useful for querying agents, pulling alerts, or managing configurations programmatically

**Questions & Answers:**
1. What is the name of the standard Linux tool that we can use to make requests to the Wazuh management server?  
   **Answer:** curl

2. What HTTP method would we use to retrieve information for a Wazuh management server API?  
   **Answer:** GET

3. What HTTP method would we use to perform an action on a Wazuh management server API?  
   **Answer:** PUT

4. Use the API console to find the Wazuh server’s version.  
   **Answer:** v4.2.5

## Final Notes & Personal Reflection

Completing this Wazuh room gave me a much stronger understanding of how a modern open-source EDR and SIEM platform actually works in practice. I now feel comfortable with the full manager-agent architecture, deploying agents, collecting logs from both Windows (via Sysmon) and Linux (via auditd and application logs), running vulnerability scans, performing policy audits, and even using the API.

- The clean dashboard visualizations and report generation make it ready for real SOC use.

**Areas I want to explore further:**

- Deploying Wazuh in a small home lab or test environment with multiple agents.
- Looking into production best practices like clustering, indexer tuning, and long-term log retention.

Overall, Wazuh feels like a legitimate enterprise-grade tool that happens to be completely free and open-source. It ties together many concepts from the SOC Level 1 path really well. I’d definitely consider using it for personal monitoring or even in a small organization setup.
